# Система компьютерной алгебры. Стандарт разработки.

## Полезные команды:

```bash
pytest                      # запуск тестов
pre-commit run --all-files  # запуск процессе прекоммита без самого коммита
```

## Архитектура проекта.

Проект разделен на независимые слои. UI и математика никак не зависимы друг от друга.
Структура проекта:

```
├── core        --> Ядро проекта. Только математика.
│ ├── base      --> Базовые модели данных: классы Natural, Integer и тд.
│ └── modules   --> Основные модули, реализация функций. (из задания)
├── main.py     --> Входная точка проекта.
└── ui          --> Интерфейс программы
├── components  --> Компоненты интерфейса
│ ├── dialogs   --> Диалоговые окна
│ ├── modules   --> Окна для модулей
└── views       --> Главные слои интерфейса
```

## Правила нейминга.

Математические модули: мы придерживаемся именования из [задания](https://docs.google.com/document/d/1Dv_6AIhxg_3ezu6VMcEnMpyfRzgym9l8PmE4ULGfjgM/edit?tab=t.0) (имена модулей), чтобы было легче проверять и сравнивать что уже реализованно, а что нет.

Ветки: название ветки = название модуля. Если требуется вторая версия модуля или рефакторинг модуля в конце дописываем \_refactoring или \_v{номер версии}

Название тестов. Они должны быть по след. шаблону: "test*{module/base}*{название того что тестируют}".

## Оформление функций.

Для каждой функции прописывайте описание в формате docstrings. так же прописывайте типизацию для упрощения разработки.
Пример функции:

```python
def multiply(a: int, b: int) -> int:
    """
    Умножение двух чисел.

    Args:
        a (int): Первое число.
        b (int): Второе число.

    Raises:
        ValueError: Параметры не соответствуют типам.

    Returns:
        int: Произведение чисел.
    """
    if not (isinstance(a, int) or isinstance(b, int)):
        raise ValueError("Invalid value")
    return a * b
```

## GitFlow.

Разработка ведётся по модели ветвления GitFlow, для структурирования процесса разработки и выпуска.
Основые положения:

1. Ветка main - для стабильной версии проекта.
2. Ветки feat/<название нововедения> - ветки содержащие в себе реализацию новой фичи. После завершения разработки фичи, создается ПуллРеквест и по решению codereview архитектора нововведение заливается им в main.
3. Ветка fix/<название фикса> - ветки с фиксом бага/багов.

Замечания из ПР выполнять в той же ветке!!!

Pre-commit: перед каждым коммитом запускается проверка. Если она ругается - исправьте (автофикс работает, но не на всё), иначе коммит не пройдет.
